# Plan mode je permission mode

**Plan mode** — read-only. Claude explores the codebase and proposes a plan but touches nothing until you approve it. Reach it with two Shift+Tab presses (or `/plan`). Because it's not writing files or running tools, it's fast and cheap regardless of model.

Shortcut `Shift` + `Tab` menja kroz modove koji su modovi dozvola.

ima nekoliko ovih modova, procitaj o njima nesto vie kada ih budes koristio.

## Komanda za plan mode

```shell
# u plan mode mozes uci takodje kroz
/plan
```

## Modovi koje vidiš kroz Shift+Tab ciklus

Evo pregled svih permission modova u Claude Code-u — uključujući ona četiri koja vidiš kroz Shift+Tab, plus još dva koja postoje ali se ne pojavljuju automatski u tom ciklusu.

**1. Manual mode (nekad zvan "default")** Standardni, najkonzervativniji mod. Kada Claude želi da izmeni fajl, pokrene shell komandu ili napravi network zahtev, zastaje i traži da odobriš tu akciju. Čitanje fajlova ide bez pitanja, ali sve što menja stanje sistema traži potvrdu. Najbolji izbor za osetljiv rad gde želiš da vidiš i odobriš svaku akciju pojedinačno.

**2. Accept edits mode** Auto-odobrava izmene fajlova i osnovne filesystem komande (mkdir, touch, rm, mv, cp, sed i slično) unutar tvog working directory-ja. Koristi ovaj mod kada želiš da pregledaš izmene naknadno kroz editor ili git diff, umesto da odobravaš svaku izmenu pojedinačno. Sve ostale Bash komande (npr. pokretanje testova, network pozivi) i dalje traže odobrenje. Bitna napomena — ovaj mod NE odobrava baš sve, samo edits i "bezbedne" filesystem operacije.

**3. Plan mode** Ovaj mod govori Claude-u da istraži kod i predloži izmene bez da ih zapravo napravi. Čisto read-only — Claude čita, analizira, pravi plan, ali ne dira nijedan fajl. Kad je plan gotov, Claude ga predstavlja i pita kako da nastavi — možeš odobriti plan i preći u accept edits, manual, ili nastaviti da doterujuš plan. Odličan za istraživanje nepoznatog koda pre nego što se commit-uješ na izmene.

**4. Auto mode** Najnoviji dodatak, i najautonomniji od ova četiri. Umesto da te pita za svaku akciju, rutira tool pozive kroz poseban klasifikator model koji blokira sve što eskalira izvan tvog zahteva, cilja na nepoznatu infrastrukturu, ili deluje pokrenuto od strane neprijateljskog sadržaja koji je Claude pročitao — dakle to je zaštita i od prompt injection napada. Rizične stvari poput push-a na main, deploy-a u produkciju, slanja secrets-a spolja, ili brisanja fajlova koji su postojali pre sesije i dalje bivaju blokirane. Zahteva noviju verziju Claude Code-a i određeni plan (Pro/Max/Team/Enterprise, zavisno od verzije).

Redosled ciklusa Shift+Tab je obično: **Manual → Accept edits → Plan → (Auto, ako je omogućen)**.

## Dva moda koja NISU u default ciklusu

**dontAsk mode** — samo tool-ovi koji odgovaraju tvojim permissions.allow pravilima se izvršavaju; sve ostalo se odbija bez pitanja. Podešava se samo pri startu ili preko konfiguracije, ne kroz Shift+Tab — koristan za CI pipeline-ove.

**bypassPermissions (YOLO mode)** — skida sve permission provere, Claude izvršava bilo koji tool bez pitanja. Koristiti samo u potpuno izolovanim okruženjima poput kontejnera, VM-ova ili efemernih CI runner-a gde Claude ne može napraviti trajnu štetu — čak i tu, zaštićene putanje (.git, .claude konfiguracija) i dalje traže potvrdu.

Ево — четири мода из твог Shift+Tab циклуса, поређани по томе шта дозвољавају и када их бирати.

- **Manual (default)** — svaki put pita za dozvolu pre izmene fajla ili bash komande.
- **Accept edits** — izmene fajlova prolaze automatski, bash komande i dalje traže potvrdu.

Znači git history (npr. `git rebase`, `git reset --hard`, `git push --force`) je samo **Bash komanda** — tretira se kao i svaka druga bash komanda u zavisnosti od moda, nema poseban "git mod".

# Постоји и `permissions` поље у `.claude/settings.json`


**Razlika između moda i `permissions` polja u `.claude/settings.json`:**

Mod je **grubo, sesijsko podešavanje** — jedna promenljiva koja kaže "koliko trenja ima uopšte" (pitaj za sve / pitaj samo za bash / ne diraj ništa / ne pitaj ni za šta).

`permissions` polje je **fino, trajno pravilo po alatu i šablonu (pattern)** — nezavisno od moda:

```json
{
  "permissions": {
    "allow": ["Bash(git status)", "Bash(git diff:*)"],
    "deny": ["Bash(git push --force:*)", "Read(./.env)"],
    "ask": ["Bash(git push:*)", "Bash(git rebase:*)"],
    "defaultMode": "default"
  }
}
```

Ključna stvar: **pravila iz `permissions` polja imaju prednost nad modom.** Redosled evaluacije je uvek `deny → ask → allow`, i tek ako ništa ne odgovara, koristi se trenutni mod kao fallback.

Praktičan primer — čak i ako si u **Accept-edits modu** (gde se izmene fajlova automatski odobravaju), ako imaš:

```json
"deny": ["Bash(git push --force:*)"]
```

Claude **nikad** neće moći da uradi force push, bez obzira na mod. Isto važi obrnuto — `defaultMode` unutar `permissions` objekta je zapravo način da trajno podesiš koji mod je podrazumevani za projekat (umesto da svaki put ručno prebacuješ Shift+Tab-om).

Ukratko: **mod = privremeni/sesijski nivo trenja**, **permissions = trajna, granularna pravila po komandi/fajlu koja važe bez obzira na mod** — i deny pravila iz `permissions` uvek pobeđuju, čak i ako si u modu koji bi inače sve automatski odobrio.
